# Volume Performance Groups (VPGs)

## Overview

Volume Performance Groups (VPGs) are pool-scoped performance policy groups used for manual Quality of Service (QoS) management in the VSA Control Plane. VPGs allow users to define throughput (MiBps) and IOPS limits for volumes, enabling fine-grained performance control and resource allocation within a pool.  The goal of this behavior is to allow end users to prevent bully volume workloads from taking over all of the pool's throughput and iops.

This document describes what VPGs are, how they are created, the differences between auto-generated and non-auto-generated VPGs, and provides examples of how to configure pools and volumes with performance limits.

**Reference Specification**: [Manual QoS for VSA Functional Spec](https://confluence.ngage.netapp.com/spaces/CLOUDVOL/pages/1378924129/Manual+QoS+for+VSA+Functional+Spec)

## What are Volume Performance Groups?

A Volume Performance Group (VPG) is a database entity that represents a QoS policy group in ONTAP. Each VPG contains:

- **Name**: A unique identifier for the VPG within a pool
- **PoolID**: The pool this VPG belongs to (VPGs are pool-scoped)
- **ThroughputMibps**: Maximum throughput in MiBps
- **Iops**: Maximum IOPS
- **IsShared**: Whether the volumes assigned to this VPG share the declared throughputMibps and iops between them (IsShared = true) or if they each get exclusive access to their own x throughputMibps and y iops.
- **IsAutoGen**: Whether the VPG was auto-generated by the system (If it's used for an individual volume's QoS autogenerated will be true)
- **OntapQosPolicyID**: The UUID of the corresponding QoS policy group in ONTAP

VPGs map to ONTAP QoS policy groups (QPGs), which enforce the performance limits at the storage layer. When a volume is assigned to a VPG, the volume's Qos Policy Group (QPG) in ONTAP is set to the VPG's corresponding QPG.  While every VPG will have a corresponding QPG, not every QPG in an ONTAP cluster used by VCP will have a corresponding VPG.

**Code Reference**: See `VolumePerformanceGroup` struct definition in `core/datamodel/models.go` (lines 331-343).

## How VPGs are Created

VPGs can be created in two ways:

### 1. Non-Auto-Generated (Manual) VPGs

Manual VPGs are created through the VPG API endpoints (when enableVpgEndpoints is true).

**Code Reference**: See `CreateVolumePerformanceGroup` function in `core/orchestrator/volume_performance_group.go` (lines 14-17).

### 2. Auto-Generated VPGs

Auto-generated VPGs are created automatically when a volume is created or updated to have individual QoS values (`throughputMibps` and optionally `iops`) specified.
They are not considered when interacting with the VPG endpoints (autogenerated VPGs will not show up in ListVPGs response) as identifying autogenerated VPGs is the approach that allows end users to specify throughput and QoS per volume without needing to first manually create a VPG for each volume that will only ever include that volume.


The system:

1. Calculates IOPS if not provided (when enableInferredIops is true)
2. Creates a QoS Policy Group (QPG) in ONTAP first
3. Creates the VPG in the database with the QPG UUID
4. Assigns the volume to the VPG

**Code Reference**: See auto-generated VPG creation logic in `core/orchestrator/workflows/volume_create_workflow.go` (lines 1020-1067).

**IOPS Calculation**: When IOPS inference is enabled, IOPS is calculated proportionally based on the pool's total throughput and IOPS using the formula: `iops = floor(total_iops * throughput / total_throughput)`.

**Code References**:
- See `calculateIopsFromThroughput` function in `core/orchestrator/volume.go` (lines 119-121).
- See IOPS calculation usage in `core/orchestrator/volume.go` (lines 483-488).




## Differences Between Auto-Generated and Non-Auto-Generated VPGs

| Feature | Auto-Generated VPGs | Non-Auto-Generated VPGs |
|---------|---------------------|-------------------------|
| **Creation** | Created automatically when volume is created with `throughputMibps` | Created manually via API |
| **Naming** | Format: `autoGenerated-{volumeName}-{volumeUUID}` | User-specified name |
| **IsAutoGen Flag** | `true` | `false` |
| **Lifespan** | Automatically cleaned up when last volume using it is deleted | Must be manually deleted |
| **Assignment** | Automatically assigned to the volume that triggered creation | Volumes can be assigned via `volumePerformanceGroupId` parameter |
| **Update Behavior** | Can be updated in-place (only 1 QoS policy to be updated in ONTAP) | Update behavior differs based on vpg.IsShared state and number of volumes assigned. |
| **Sharing** | Has no effect on single volume VPG (`IsShared=false` by default) | Can be shared (configurable) |
| **Assignment Restrictions** | Cannot be assigned to volumes via `volumePerformanceGroupId` | Can be assigned to volumes via `volumePerformanceGroupId` |

**Code Reference - Auto-Generated VPG Cleanup**: See `CleanupAutoGeneratedVPG` function in `core/orchestrator/activities/volume_delete_activities.go` (lines 70-112).


## VPG Usage in Code

### Volume Creation with VPG

When creating a volume, you can either:
1. Specify `throughputMibps` and  `iops` to auto-generate a VPG
2. Specify `volumePerformanceGroupId` to assign to an existing VPG

**Code Reference**: See volume creation with VPG logic in `core/orchestrator/volume.go` (lines 478-518).

### Volume Update with VPG

Volumes that are within a manual qosType pool can be updated to:
1. Change their throughput/IOPS (if assigned to an auto-generated VPG, the VPG is updated in-place)
2. Assign to a different VPG

**Code Reference - Updating Auto-Generated VPG**: See in-place VPG update logic in `core/orchestrator/workflows/volume_update_workflow.go` (lines 540-567).

**Code Reference - Assigning to Different VPG**: See VPG reassignment logic in `core/orchestrator/workflows/volume_update_workflow.go` (lines 627-685).

### Pool Capacity Validation

When volumes are created or updated with throughput/IOPS in a manual qosType pool, the system validates that the sum of all volumes' throughput and IOPS does not exceed the pool's total capacity.

**Code Reference**: See `validatePoolCapacityForVolume` function in `core/orchestrator/volume.go` (lines 1192-1271).

**Shared VPG Handling**: For shared VPGs, the capacity is only counted once even if multiple volumes use the same VPG. See `shouldSubtractCurrentVpgContribution` function in `core/orchestrator/volume.go` (lines 1273-1292).

## Examples: Setting Up Pools with Volume Performance Limits

The following examples demonstrate how to configure pools and volumes so that one or more volumes are limited to 40% of the pool's throughput and IOPS.

### Prerequisites

- MQOS (Manual QoS) must be enabled
- Pool must have `qosType` set to `manual`
- Pool must have custom performance attributes set (throughput and IOPS)

### Example 1: Single Volume with 40% Limit

**Scenario**: Create a pool with 1000 MiBps throughput and 16000 IOPS, then create a volume limited to 40% (400 MiBps, 6400 IOPS).

**Step 1: Create Pool with Manual QoS**

```json
POST /v1beta/projects/{project}/locations/{location}/pools
{
  "pool": {
    "name": "my-pool",
    "qosType": "manual",
    "throughputMibps": 1000,
    "iops": 16000,
    "quotaInBytes": "1099511627776"  // 1 TiB
  }
}
```

**Step 2: Create Volume with 40% Performance Limit**

When IOPS inference is enabled:
```json
POST /v1beta/projects/{project}/locations/{location}/volumes
{
  "volume": {
    "name": "limited-volume",
    "poolId": "pool-uuid",
    "quotaInBytes": "107374182400",  // 100 GiB
    "throughputMibps": 400
  }
}
```

When IOPS inference is disabled:
```json
POST /v1beta/projects/{project}/locations/{location}/volumes
{
  "volume": {
    "name": "limited-volume",
    "poolId": "pool-uuid",
    "quotaInBytes": "107374182400",  // 100 GiB
    "throughputMibps": 400,
    "iops": 6400
  }
}
```

**What Happens**:
1. System calculates IOPS (if inference enabled): `floor(16000 * 400 / 1000) = 6400 IOPS`
2. Auto-generated VPG is created with name `autoGenerated-limited-volume-{uuid}`
3. QoS policy is created in ONTAP with 400 MiBps and 6400 IOPS
4. Volume is assigned to the VPG

**Code Path**: The volume creation flow in `core/orchestrator/volume.go` (lines 478-518) handles this scenario.

### Example 2: Multiple Volumes Sharing 40% Limit

**Scenario**: Create a pool with 1000 MiBps throughput and 16000 IOPS, then create two volumes that share a 40% limit (400 MiBps, 6400 IOPS total, shared between them).

**Step 1: Create Pool with Manual QoS**

Same as Example 1.

**Step 2. Create VPG with 40% Performance Limit and isShared=False**

This will create a VPG with isAutoGenerated = false, meaning that volumes can be assigned to it.

**Step 2: Create First Volume and Assign to VPG**

First, list VPGs to get the VPG UUID:
```json
GET /v1beta/projects/{project}/locations/{location}/pools/{poolId}/volumePerformanceGroups
```

Then create the first volume and assign it to the existing VPG:
```json
POST /v1beta/projects/{project}/locations/{location}/volumes
{
  "volume": {
    "name": "limited-volume-1",
    "poolId": "pool-uuid",
    "quotaInBytes": "107374182400",  // 100 GiB
    "volumePerformanceGroupId": "vpg-uuid-from-step-2"
  }
}
```

**Step 3: Create Second Volume and Assign to Same VPG**

Then create the second volume and assign it to the existing VPG:
```json
POST /v1beta/projects/{project}/locations/{location}/volumes
{
  "volume": {
    "name": "limited-volume-2",
    "poolId": "pool-uuid",
    "quotaInBytes": "107374182400",  // 100 GiB
    "volumePerformanceGroupId": "vpg-uuid-from-step-2"
  }
}
```

**What Happens**:
1. First volume creates VPG with 400 MiBps and 6400 IOPS
2. Second volume is assigned to the same VPG
3. Both volumes share the 400 MiBps / 6400 IOPS limit (not 400 MiBps each)
4. Pool capacity shows 400 MiBps / 6400 IOPS used (not 800 MiBps / 12800 IOPS)

**Code Path**: The volume assignment logic in `core/orchestrator/volume.go` (lines 500-518) validates and assigns volumes to existing VPGs.

**Important Note**: For shared VPGs, the capacity is counted once in pool utilization calculations, as shown in `core/orchestrator/volume.go` (lines 1273-1292).

### Example 3: Multiple Volumes Each with 40% Limit

**Scenario**: Create a pool with 1000 MiBps throughput and 16000 IOPS, then create two volumes each limited to 40% (400 MiBps, 6400 IOPS each).

**Step 1: Create Pool with Manual QoS**

Same as Example 1.

**Step 2: Create First Volume with 40% Performance Limit**

Same as Example 1, Step 2.

**Step 3: Create Second Volume with 40% Performance Limit**

```json
POST /v1beta/projects/{project}/locations/{location}/volumes
{
  "volume": {
    "name": "limited-volume-2",
    "poolId": "pool-uuid",
    "quotaInBytes": "107374182400",  // 100 GiB
    "throughputMibps": 400
  }
}
```

**What Happens**:
1. First volume creates VPG #1 with 400 MiBps and 6400 IOPS
2. Second volume creates VPG #2 with 400 MiBps and 6400 IOPS
3. Each volume has its own VPG and 40% limit
4. Pool capacity shows 800 MiBps / 12800 IOPS used (80% of pool capacity)
5. Pool validation ensures total doesn't exceed pool limits

**Code Path**: Pool capacity validation in `core/orchestrator/volume.go` (lines 1192-1271) ensures the sum doesn't exceed pool totals.

### Example 4: Updating Volume Performance Limits

**Scenario**: Update an existing volume's performance limit to 40% of pool capacity.

**Step 1: Update Volume with New Throughput/IOPS**

If the volume has an auto-generated VPG:
```json
PATCH /v1beta/projects/{project}/locations/{location}/volumes/{volumeId}
{
  "throughputMibps": 400,
  "iops": 6400
}
```

**What Happens**:
1. If volume has auto-generated VPG, the VPG is updated in-place
2. QoS policy in ONTAP is updated with new values
3. VPG database record is updated
4. Pool capacity validation ensures new total doesn't exceed pool limits

**Code Path**: The update workflow in `core/orchestrator/workflows/volume_update_workflow.go` (lines 540-567) handles in-place updates for auto-generated VPGs.

**Step 2: Update a manual VPG via Update Volume Performance Group (two volumes, isShared=false)**

Consider a manual VPG (`isAutoGenerated=false`, `isShared=false`) with two volumes assigned. Updating that VPG's throughput via the VPG update API:

```json
PATCH /v1beta/projects/{project}/locations/{location}/pools/{poolId}/volumePerformanceGroups/{volumePerformanceGroupId}
{
  "throughputMibps": 300,
  "iops": 4800
}
```

**What Happens**:
1. Pool capacity validation counts this VPG once per assigned volume: 2 × 300 = 600 MiBps and 2 × 4800 = 9600 IOPS. The new total for the pool must not exceed the pool's capacity.
2. The VPG's single QoS policy group (QPG) in ONTAP is updated with the new throughput and IOPS.
3. The VPG record in the database is updated. No per-volume DB changes are required; both volumes already point to this VPG.
4. After the update, both volumes immediately see the new per-volume limit from ONTAP.

**Important**: With `isShared=false`, increasing the VPG's throughput/IOPS increases the **per-volume** limit for every volume in the group, so pool usage for this VPG is (throughputMibps × number of volumes) and (iops × number of volumes). A large increase can therefore fail pool capacity validation if the pool would be overcommitted.

**Code Path**: `UpdateVolumePerformanceGroup` in `core/orchestrator/volume_performance_group.go` loads the VPG, validates pool capacity (accounting for all volumes in the VPG and `isShared`), updates the QPG in ONTAP, then updates the VPG in the database. The endpoint handler is in `google-proxy/api/endpoints/volume_performance_group_endpoints.go` (V1betaUpdateVolumePerformanceGroup);

## Feature Flags and Configuration

VPG functionality is controlled by feature flags:

- **MQOS**: Must be enabled for volume-level throughput/IOPS
- **ENABLE_VPG_ENDPOINTS**: Controls VPG API endpoints (list, describe, create, update, delete)
- **ENABLE_VOLUME_PERFORMANCE_GROUP_ASSIGNMENT**: Controls ability to assign volumes to existing VPGs via `volumePerformanceGroupId`
- **ENABLE_INFERRED_IOPS**: Controls automatic IOPS calculation from throughput

**Code Reference**: See feature flag definitions in `google-proxy/api/endpoints/volume_performance_group_endpoints.go` (lines 16-18).

**ConfigMap Reference**: See ConfigMap template in `kubernetes/vsa-control-plane/charts/google-proxy/templates/configMap.yaml` (lines 148-149).

## Validation Rules

### Volume Creation/Update Validation

1. **Mutually Exclusive Parameters**: Cannot specify both `throughputMibps` and `volumePerformanceGroupId`
2. **Pool Type**: Auto pools reject `throughputMibps` and `iops`; manual pools require `throughputMibps` when MQOS is enabled
3. **IOPS Requirement**: When `enableInferredIops` is false, `iops` must be provided with `throughputMibps`
4. **VPG Assignment**: Cannot assign volumes to auto-generated VPGs
5. **Pool Validation**: VPG must belong to the same pool as the volume
6. **Capacity Validation**: Sum of all volumes' throughput/IOPS cannot exceed pool totals
7. **Pool Update Validation**: All updates to pool size and available qos will be verified such that they don't drop capacity below what is allocated to child volumes.
8. **VPG Deletion**: Checks that no volumes are assigned to a VPG before deleting it.

**Code Reference**: See validation logic in `core/orchestrator/volume.go` (lines 1474-1572).

## Database Schema

VPGs are stored in the `volume_performance_groups` table with the following key fields:

- `id`: Primary key
- `uuid`: Unique identifier
- `name`: VPG name
- `pool_id`: Foreign key to pools table
- `is_shared`: Boolean indicating if VPG can be shared
- `is_auto_gen`: Boolean indicating if VPG was auto-generated
- `throughput_mibps`: Maximum throughput in MiBps
- `iops`: Maximum IOPS
- `ontap_qos_policy_id`: ONTAP QoS policy group UUID

**Code Reference**: See `VolumePerformanceGroup` struct in `core/datamodel/models.go` (lines 331-343).
