# Cluster Serial Number Generation Design

## Context

The VSA control plane needs to assign a globally unique serial number to every node it provisions. The serial number must:
- Be unique across all clusters, regions.
- Be scalable to very high creation rates (e.g., 150,000 clusters per day).
- Avoid reliance on any central service or in-memory state for uniqueness.
- Be compatible with downstream systems (VLM) that require a 20-digit node serial number.

## Decision


We generate a 20-digit serial number for each cluster. The structure is:
- **3-digit fixed prefix**: e.g., `935` (configurable via environment variable `CLUSTER_SERIAL_NUMBER_PREFIX`).
- **2-digit region code**: Each region is mapped to a unique 2-digit code (see `REGION_NUMBER_MAP`). Supports up to 99 regions (currently 42 in use).
- **15-digit per-region counter**: Each region maintains its own monotonically increasing counter, persisted in the database.

This approach provides 1 quadrillion (10^15) unique serials per region, and with 42 regions, the system can safely generate unique serials for **over 18,264 years** at a rate of 150,000 clusters per day, per region.

## Consequences

- **No central service or memory required**: Each region's counter is independent and persisted, so there is no need for a global lock, distributed cache, or central authority.
- **Scalability**: The design supports extremely high rates of cluster creation without risk of collision or bottleneck.
- **Simplicity**: The logic is simple, auditable, and easy to reason about.
- **Downstream compatibility**: The 20-digit serial number meets the requirement for VLM node serials. All 20 digits are generated by the control plane; there is no reservation for VLM.

## Example

For region `us-central1` (code `34`), the first cluster might get serial number `935340000000000000001`. All 20 digits are generated and assigned by the control plane.
